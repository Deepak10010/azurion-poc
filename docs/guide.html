A friendly guide

The robot that helps doctors see inside you.

This project is a pretend hospital X-ray robot you can play with on a computer — plus a set of robot helpers that check it works. It's a safe place to learn a real job: testing machines that move.

01  What is this, really?

A giant camera that takes photos from the inside

In hospitals there is a machine called the Azurion. Think of it as a huge robot arm shaped like the letter C. On one end is an X-ray camera; on the other end is the X-ray light. The patient lies on a table in the middle.

The whole C can spin, tilt, and slide, and the table moves too. That lets the doctor look at a heart or a blood vessel from any angle — without cutting the patient open. Instead they slide a tiny tube through a blood vessel and watch it on screen, live.

🎥

Picture a film crew

The C-arm is the camera on a crane. The patient is the actor. The doctor is the director saying "move left, tilt down, zoom in." The machine must obey exactly — every time — because a real person is lying right there.

Our project doesn't use a real machine. It uses a simulator: a software copy that follows the same rules — it moves at the same speed, has the same limits, and can even bump into things. You can watch it move in your web browser.

02  Why do we need it?

Because checking a moving machine is surprisingly tricky

Before a hospital trusts the machine, someone has to test it: give it a command and check it did the right thing. We want a computer to do that checking automatically, thousands of times. That's called test automation.

Testing a website is easy — buttons appear instantly. But a moving machine takes time. And that small fact causes a big, sneaky problem.

🍞

The toaster problem

You push the toaster lever and check for toast half a second later. No toast! Is the toaster broken? No — you just didn't wait. A bad test makes this exact mistake: it checks too early, then shouts "BROKEN!" when nothing is wrong. Tests that fail randomly like this are called flaky, and they drive engineers crazy.

Why pretend instead of using the real one?

  • A real machine costs millions and there aren't many to practise on.
  • It uses real radiation — not something to play with while learning.
  • It's busy with real patients all day.
  • A simulator can be made flaky on purpose, so you can practise beating the toaster problem safely.

03  The one big lesson

Wait for it to finish. Then check.

This is the whole secret of testing machines that move, and it's the skill this project teaches you to feel in your hands:

⏳

The rule

Don't check while it's still moving. Wait until the machine says "I've arrived and settled down," and then check that it landed close enough to where you asked (a tiny bit of wobble is normal — just like a real motor).

In this project there's a flakiness switch with three settings: Off, Mild, and Harsh. Turn it to Harsh and the pretend machine wobbles and overshoots like a tired robot. The good tests still pass — because they wait and allow for wobble. That's the proof the lesson works.

04  What's inside the box

Three parts that work together

PART 1

The pretend machine

The software robot itself. It knows how to move each part, how fast, how far, and when it would crash. This is the thing we test.

PART 2

The live screen

A web page that draws the machine and updates as it moves — so you can watch the arm swing and the table rise in real time.

PART 3

The robot helpers (tests)

Little scripts written in almost-plain-English that command the machine and check it did the right thing — automatically.

The tests are written in a style called Gherkin that reads like a story. Here's a real one from this project:

Positioning.feature 
Scenario: Rotate the stand to a single oblique angle
  When I command StandRotation to 30 degrees
  And I wait for the system to settle
  Then StandRotation is at 30 within 0.05
  And no fault is raised

See it? Command → wait → then check. That's the toaster rule, written down.

05  How to run it · step by step

Let's make it go

You'll type a few commands into a terminal (a text window where you talk to your computer). Don't worry — you only need to copy and paste.

First, install one thing: the .NET 10 SDK from dotnet.microsoft.com/download. To check it worked, type dotnet --version — you should see a number starting with 10.
Path A · just check everything works
1

Open the project folder in a terminal

Open the folder c:\Repos\Azurion in a terminal window.

2

Run the tests

Type this and press Enter:

terminal
> dotnet test

The computer quietly builds the pretend machine in the background and runs every robot helper against it. Wait about a minute (the tests really do wait for the machine to move). At the end you want to see:

result
Passed!  - Failed: 0, Passed: 26, Skipped: 0, Total: 26

That's it — every check passed. 🎉

Path B · watch the machine move (the fun one)
1

Start the machine + screen

In your first terminal, type:

terminal 1
> dotnet run --project src/Azurion.Web

Leave this one running.

2

Open the live screen in your browser

Go to this address:

browser
http://localhost:5023

You'll see the C-arm, the table, and a panel of numbers. Try the jog buttons!

3

Run the tests in a second terminal

Open another terminal in the same folder and type:

terminal 2
> dotnet test

Now look at your browser — the arm moves by itself as the tests drive it. You're watching robot helpers operate the machine, live.

4

Turn on the wobble

On the screen, set Flakiness → Harsh and run dotnet test again. Watch the arm overshoot and drift — yet the good tests still pass, because they wait and allow for wobble. That's the whole lesson, in action.

06  Learn by playing

Little experiments to try

The best way to learn is to poke at it and see what happens. Nothing here can break anything — it's all pretend. Try these:

EXPERIMENT 01

Break a test on purpose

Open a .feature file and change within 0.05 to within 0.001 under Harsh wobble. Watch it go flaky — then change it back.

EXPERIMENT 02

Cause a crash

On the screen, press "Force collision." The arm and table get too close, the machine shouts a warning, and motion stops. Safety in action.

EXPERIMENT 03

Ask for the impossible

Command an axis past its limit (way too far). The machine refuses and raises a fault instead of breaking itself.

EXPERIMENT 04

Write your own helper

Copy a scenario in Positioning.feature, change the angle, and run the tests. You just wrote a test!

07  Word decoder

Tricky words, in plain language

Hospitals and software both love their special words. None of them are scary once you know what they really mean. Here's the whole list — keep it open while you play.

The machine and its parts

C-arm
The big robot arm shaped like the letter C. One tip holds the X-ray camera, the other holds the X-ray light. The patient lies in the gap between them.
Geometry
A fancy word for "where everything is right now" — how the arm is pointing and where the table sits. Changing the geometry just means moving things to look from a new angle.
Axis (many: axes)
One single way the machine can move — like one slider or dial. Spinning is one axis, tilting is another, the table going up is another. Each one moves on its own.
Isocenter
The fixed point in the very middle that the arm always aims at. However the arm spins, it keeps pointing here — where the doctor wants to look. It's the little yellow cross on the screen.
Source
The part that makes the X-ray beam — the "light" that shines through the patient.
Detector
The part that catches the X-rays after they pass through the patient and turns them into the picture — the "camera."
SID (source-image distance)
How far apart the light and the camera are. Sliding the camera in or out changes how big and sharp the picture looks.

The movements (angle names)

RAO / LAO
How far the arm leans to one side as it spins around the patient. RAO = leans to the patient's right, LAO = to the left. (Doctors say "Right/Left Anterior Oblique.")
LAO 30°
A command meaning "spin the arm 30 degrees over to the left side." It's just one particular viewing angle. (And ° is the little symbol for "degrees.")
CRAN / CAUD
How far the arm tilts toward the head or the feet. CRAN = toward the head, CAUD = toward the feet. (Short for "cranial / caudal.")
Roll (propeller)
Spinning the C around like an aeroplane propeller.
Table height / lateral / longitudinal
The patient table moving up-down (height), side-to-side (lateral), and head-to-feet (longitudinal).
Tilt / cradle
Tipping the whole table — head-end down (tilt) or rocking side to side (cradle).

Buttons on the screen

Home
Sends every part back to its normal starting position — the "tidy up and go to neutral" pose.
Jog
To nudge a part by hand, a little at a time, using buttons — instead of typing an exact number to go to.
Force collision
A button that deliberately drives the arm and table toward each other until they'd bump — so you can watch the safety system catch it and stop everything. (Totally safe — it's all pretend.)
Emergency stop (E-STOP)
The big red "freeze everything NOW" button. Nothing moves again until you press Reset.
Reset
Clears any warnings and puts the machine back to a clean starting state.
Flakiness (Off / Mild / Harsh)
A switch that makes the pretend machine behave more or less wobbly and unreliable on purpose — so you can practise writing tests that cope with it.

Words from testing

Settle / settled
When a moving part has fully stopped and stopped wobbling, it has "settled." Good tests wait for this before they check anything.
Tolerance
How much "close enough" you allow. "Within 0.05" means "if it lands within a tiny margin of the target, that counts as correct" — because real motors never stop on the exact number.
Fault
The machine's way of saying "something's wrong" — like being asked to move too far, or a crash risk. It refuses and shows a fault instead of hurting itself.
Scenario
One little test story: do this → wait → then check that. The project has lots of them.
Gherkin
The almost-plain-English language the test stories are written in, using the words When, And, and Then.
Simulator
The pretend machine itself: software that copies how the real one behaves, so you can practise safely.
